# Alpha Orbit — プロジェクト制約文書

このファイルはすべての開発タスクの前提となる。新しい機能を実装する前、既存機能を変更する前に、必ずこの文書を参照すること。

---

## このアプリは誰が、いつ、どう使うか

**使う人:** SAPIX 4年生の親（父親）。1人で使う。子供は使わない。

**使う場面:**
- 平日夜、子供が復習している横でスマホ片手に操作する
- テスト前日〜数日前に、テスト範囲の解き直し問題をまとめて子供に見せる
- 授業後にテキストをPDF化して問題を切り出す（週1〜2回、5分程度）

**使わない場面:**
- 子供が自分でアプリを開いて学習する（子供用UIは不要）
- 毎日30分かけてデータを入力する（そんな時間はない）
- 分析レポートを眺めて戦略を練る（必要なのは分析ではなく行動）

---

## このアプリで成績が上がるメカニズム

成績が上がる唯一の方法は「子供が間違えた問題を、できるようになるまで解き直すこと」。

アプリの役割は以下の2つだけ:

1. **間違えた問題を記録する** — PDFテキストから問題を切り出してクリップする
2. **必要なときに引っ張り出す** — テスト範囲で絞り込んで、間違い問題を画像付きで一覧表示する

これ以外の機能は「あると便利かもしれない」であって「ないと困る」ではない。

## 開発の判断基準

### 機能を追加する前に問うこと

1. **その機能がないと、子供の解き直しが止まるか？** → No なら作らない
2. **親の入力時間が増えるか？** → 増えるなら、増える時間に見合う価値があるか検証する
3. **既存の機能で代替できないか？** → できるなら作らない

### やってはいけないこと

- **分析ダッシュボードを増やす。** 習熟度スコア、偏差値推移、正答率グラフは「見て満足する」だけで行動につながらない。
- **自動推薦エンジンを作る。** 「おすすめ復習」のように、アルゴリズムが何を復習すべきか提案する機能は不要。何をやるべきかは親が判断する。アプリは判断材料（間違い問題リスト）を出すだけ。
- **入力ステップを増やす。** 問題登録時にmissType（理解不足/ケアレス）、難易度、詳細メモを求めない。問題番号と画像だけで十分。
- **子供向けUIを作る。** ゲーミフィケーション、ポイント、バッジは不要。
- **仕様にないものを勝手に追加する。** 「あると便利そう」で機能を増やさない。仕様書に書かれていないことは実装しない。

### UIの原則

- **片手スマホ操作が前提。** ボタンは大きく、タップ領域は十分に確保する。
- **最短経路を優先。** 「問題を切り出す」「テスト範囲の問題を見る」の2つの操作が最短ステップで完了すること。
- **情報は最小限。** 画面に出す情報は、その画面で行動するために必要なものだけ。missType、正答率、習熟度パーセントは表示しない（ノイズになる）。
- **選択肢を減らす。** 「全部できた / 止まった問題があった」のような2択は許容。3択以上は疑う。設定画面のオプションは増やさない。

## 既存機能の位置づけ

| 機能 | 状態 | 役割 |
|---|---|---|
| テキスト管理 + PDF | ✅ 核心機能 | テキストをPDF化して保存。問題切り出しの入口 |
| 問題クリッパー（PdfCropper + ProblemClipList） | ✅ 核心機能 | 間違えた問題を画像付きで切り出して保存 |
| テスト管理 + 範囲設定 | ✅ 核心機能 | テスト範囲（41B-01〜41B-05等）を登録 |
| テスト範囲の間違い問題まとめ | 🔨 未実装 | テスト範囲で問題を絞り込んで一覧表示（最優先で実装すべき） |
| 今日の家庭学習 | ⚠️ 補助 | チェックリストとしては使える。ただし核心機能ではない |
| 解き直し待ち | ⚠️ 補助 | 問題一覧として使える。ただし核心機能ではない |
| 今週の学習スケジュール | ⚠️ 補助 | 週間俯瞰。あってもなくても成績に影響しない |
| 単元マップ | ❌ 過剰 | 50単元の習熟度一覧は4年生の段階では不要 |
| 成績（偏差値推移） | ❌ 過剰 | 偏差値グラフを見ても行動は変わらない |
| 過去問管理 | ⏸️ 時期尚早 | 6年生で使う。今は空でよい |
| カレンダー | ⚠️ 補助 | 授業日程の確認用。核心機能ではない |

## 技術的な制約

- **Firestore のスキーマを安易に変更しない。** 既存データとの互換性を維持する。
- **新しいnpmパッケージを追加しない。** 既存の依存関係で実装する。
- **コンポーネントを分割しすぎない。** 1機能 = 1コンポーネント + 1CSSが基本。ヘルパーコンポーネントの増殖を避ける。
- **既存のユーティリティ関数を再利用する。** problems.js, sapixSchedule.js, sapixTexts.js の既存関数でできることを、新しい関数で再発明しない。
